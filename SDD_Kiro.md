# Kiro SDD Template
*Основано на официальной документации [Kiro Specs](https://kiro.dev/docs/specs/)*

Этот документ описывает методологию **Kiro Specs** как вариант Spec-Driven Development, в котором работа над фичей проходит через три согласованные фазы: **requirements**, **design** и **tasks**. Подход ориентирован на пошаговое уточнение требований, проектирование решения и подготовку плана реализации до начала основной разработки.

---

## Основные артефакты Kiro

В Kiro для каждой фичи или проекта используется только одна базовая структура:

```text
specs/
  <project_name>/
    requirements.md
    design.md
    tasks.md
```

### Назначение артефактов

1. **requirements.md**
   * *Назначение:* описывает **что** нужно реализовать и **какую ценность** это дает.
   * *Содержание:* пользовательские истории, функциональные требования, acceptance criteria, edge cases.

2. **design.md**
   * *Назначение:* описывает **как** решение будет устроено технически.
   * *Содержание:* архитектура, ключевые компоненты, модели данных, интерфейсы, макеты UI, ограничения, trade-offs.

3. **tasks.md**
   * *Назначение:* разбивает дизайн на конкретные шаги реализации.
   * *Содержание:* список задач и подзадач, привязанных к требованиям и дизайну.

---

## 1. Workflow Kiro

Kiro использует последовательный процесс из трех фаз:

1. Сначала формируется `requirements.md`.
2. После согласования требований создается `design.md`.
3. После согласования дизайна создается `tasks.md`.
4. Реализация начинается уже по задачам из `tasks.md`.

### Ключевой принцип
Каждая следующая фаза опирается на предыдущую. Нельзя качественно подготовить дизайн без требований, а задачи без дизайна становятся слишком абстрактными или неполными.

---

## 2. Requirements
*Артефакт: `specs/<project_name>/requirements.md`*

**Цель:** зафиксировать ожидания от фичи на уровне поведения системы и пользы для пользователя.

### Что должно быть в requirements.md
* краткое описание фичи;
* пользовательские истории;
* функциональные требования;
* acceptance criteria;
* граничные случаи и ограничения;
* явные assumptions, если часть деталей пока не определена.

### Рекомендуемая структура

```markdown
# Requirements

## Overview
[Краткое описание фичи]

## User Stories
- As a [role], I want [goal], so that [benefit].

## Requirements
### Requirement 1
[Описание требования]

#### Acceptance Criteria
- [Критерий 1]
- [Критерий 2]

## Edge Cases
- [Случай 1]
- [Случай 2]
```

### На что делать акцент
* Формулировать требования с точки зрения наблюдаемого поведения.
* Не смешивать бизнес-ожидания с деталями реализации.
* Выявлять не только happy path, но и ошибки, пустые состояния, ограничения доступа и валидацию.

---

## 3. Design
*Артефакт: `specs/<project_name>/design.md`*

**Цель:** перевести требования в техническое решение, достаточное для уверенной реализации.

### Что обычно включает design.md
* обзор решения;
* архитектурный подход;
* основные компоненты;
* взаимодействие компонентов;
* модели данных;
* API или интерфейсы;
* макеты интерфейсов, экранов или пользовательских потоков, если фича затрагивает UI;
* решения по валидации, ошибкам, безопасности и производительности;
* технические компромиссы и ограничения.

### Рекомендуемая структура

```markdown
# Design

## Overview
[Общее описание решения]

## Architecture
[Архитектурный подход]

## Components
- [Компонент 1]
- [Компонент 2]

## Data Model
- [Сущность 1]
- [Сущность 2]

## Interfaces
- [API / контракт / интеграция]

## UI Mockups
- [Экран 1 / поток 1]
- [Экран 2 / состояние 2]

## Risks and Trade-offs
- [Риск или компромисс]
```

### Практические рекомендации
* Указывать, какие требования покрываются каждым блоком дизайна.
* Фиксировать архитектурные решения так, чтобы по документу можно было начать реализацию без догадок.
* Если изменение затрагивает интерфейс, добавлять в `design.md` макеты, wireframes, описания экранов или состояний UI.
* Явно отмечать места, где нужны компромиссы между скоростью, сложностью, поддерживаемостью и производительностью.

---

## 4. Tasks
*Артефакт: `specs/<project_name>/tasks.md`*

**Цель:** превратить дизайн в последовательный и проверяемый план работ.

### Принципы хорошего tasks.md
* задачи должны быть конкретными и выполнимыми;
* крупные шаги стоит делить на подзадачи;
* порядок должен отражать реальные зависимости;
* каждая задача должна вести к проверяемому результату;
* задачи должны ссылаться на требования или части дизайна, если это важно для трассируемости.

### Рекомендуемая структура

```markdown
# Tasks

- [ ] 1. Подготовить базовую структуру решения
- [ ] 2. Реализовать модели данных и валидацию
- [ ] 3. Реализовать основной сервисный слой
- [ ] 4. Добавить пользовательский интерфейс или API
- [ ] 5. Покрыть решение тестами
- [ ] 6. Проверить acceptance criteria
```

### Что важно
* Не писать слишком общие задачи вроде "сделать backend".
* Не дробить план до микрошагов, которые мешают видеть общую картину.
* Учитывать не только код, но и тесты, миграции, observability и интеграционные изменения, если они нужны для завершенности фичи.

---

## 5. Связь между документами

Документы Kiro образуют цепочку:

* `requirements.md` отвечает на вопрос: **что нужно сделать**;
* `design.md` отвечает на вопрос: **как это будет устроено**;
* `tasks.md` отвечает на вопрос: **в каком порядке это реализовать**.

Если в ходе проектирования выясняется, что требование неполное или противоречивое, нужно возвращаться к `requirements.md`, а не пытаться компенсировать это в `design.md` или `tasks.md`.

---

## 6. Когда использовать Kiro

Подход Kiro особенно полезен, если:
* фича не тривиальна и требует явной декомпозиции;
* над задачей работают несколько участников или LLM-агенты;
* важно заранее согласовать требования до реализации;
* нужен прозрачный переход от идеи к задачам.

Для совсем маленьких изменений такой процесс может быть избыточным, но для средних и крупных фич он помогает сократить неясность и уменьшить количество переделок.

---

## 7. Краткий шаблон Kiro

### Шаг 1. Создать структуру

```text
specs/
  my-feature/
    requirements.md
    design.md
    tasks.md
```

### Шаг 2. Заполнить requirements.md
* Что делает фича
* Для кого она нужна
* Какие есть критерии приемки

### Шаг 3. Подготовить design.md
* Как устроена реализация
* Какие компоненты и данные участвуют
* Какие есть технические риски

### Шаг 4. Сформировать tasks.md
* Какие шаги нужны для реализации
* В каком порядке выполнять работу
* Какие шаги нужны для тестирования и проверки

---

## 8. Итог

Kiro Specs опирается на простую и строгую структуру из трех документов: `requirements.md`, `design.md`, `tasks.md`. Сильная сторона методологии в том, что она не перегружает процесс лишними артефактами, но при этом сохраняет четкую связь между ожиданиями бизнеса, техническим решением и планом реализации.
