## Зачем сейчас
**Контекст (Why):**
- Нужен показательный, но компактный пример полного цикла: от требований к задачам.
- Нужен материал для обучения команды формулированию acceptance-criteria и рисков.

**Цели (Goals):**
- Создать простое десктоп-приложение “Заметки” (GUI + SQLite).
- Показать на его примере корректную структуру и связность артефактов OpenSpec.

## Область и влияние
- **Входит:**
  - GUI: список заметок, просмотр/редактирование, поиск, теги, закрепление (pin)
  - Хранилище: SQLite, схема данных, миграции схемы на уровне приложения
  - Экспорт/импорт: JSON файл
- **Не входит:**
  - Синхронизация/облако/аккаунты
  - Сетевое API и многопользовательский режим

## Критерии успеха (Acceptance)
- Можно создать, отредактировать, удалить заметку; изменения сохраняются в SQLite и видны после перезапуска.
- Поиск находит заметки по подстроке в заголовке или тексте с обновлением списка в течение 200 мс при базе до 5 000 заметок на обычном ноутбуке.
- Теги можно назначать/снимать; фильтр по тегу корректно меняет список.
- Экспорт в JSON и импорт из JSON работают без потери данных (кроме внутренних полей, описанных в specs).
- Поврежденная или недоступная база не приводит к падению приложения: показывается понятная ошибка и предлагается создать новую базу.

## Non-Goals (Не-цели)
- Шифрование базы и управление ключами.
- Совместное редактирование и конфликт-резолвинг.
- Продвинутая разметка (Markdown-рендеринг, вложения, изображения).

## Риски (высокий уровень)
- GUI-слои часто смешивают бизнес-логику с отображением → риск плохой тестируемости.
- SQLite-файл может быть заблокирован/поврежден → нужен безопасный режим обработки ошибок.

## Outcomes
- Набор артефактов OpenSpec с реалистичными требованиями, дизайном и задачами.
